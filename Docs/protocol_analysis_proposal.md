# 面向通用网络协议的多智能体证据绑定协议分析框架

## 一、题目

**中文题目**  
面向通用网络协议的多智能体、证据绑定与在线自适应分析框架设计与实现

**英文题目**  
Design and Implementation of a Multi-Agent, Evidence-Bound, Online-Adaptive Analysis Framework for General Network Protocols

---

## 二、研究背景与问题提出

HTTP、FTP、SMTP、RTSP 等通用网络协议广泛存在于真实系统中。协议分析通常涉及消息类型识别、字段约束提取、状态转移关系恢复、异常行为检测等任务。传统方法主要依赖人工阅读协议文档、静态分析协议实现、动态抓包分析或基于模糊测试的行为探索。这些方法要么人工成本高，要么对协议文档质量、实现复杂度和测试环境依赖较强。

近年来，大语言模型被引入协议分析领域。一类工作利用大语言模型从协议知识中抽取 grammar，辅助协议模糊测试；另一类工作利用大语言模型理解协议实现代码并推断状态机。这些方法已经证明 AI 对协议分析是有帮助的，但仍然存在几个共性问题：

- 分析过程高度依赖自然语言推理，中间过程难以约束，结果不稳定
- 结论往往缺少清晰证据绑定，难以追踪每条状态转移和约束的来源
- 很多方案偏重 fuzzing 或单次离线推断，缺少边交互边修正的分析闭环
- 系统输出多为离线文本或静态图，缺少对分析过程和证据链的可视化展示

因此，本课题拟设计一个**面向通用网络协议的多智能体协议分析框架**。该框架不特化于 AI agent 协议，而是以 FTP、SMTP、HTTP 子集等典型网络协议为对象，结合多智能体协作、function calling、证据绑定与在线探测机制，构建一个可解释、可修正、可视化的通用协议分析系统。

---

## 三、研究目标

本课题拟实现一个面向通用协议的智能分析系统，具备以下能力：

1. 从协议文档、抓包 trace、在线交互日志等多源输入中提取协议知识
2. 自动识别消息类型、字段结构、字段约束和顺序依赖
3. 构建候选协议状态机，并对状态转移给出置信度和证据来源
4. 通过在线探测对歧义状态和可疑转移进行判别与修正
5. 通过 WebUI 展示协议结构、状态图、证据链和在线修正过程

本课题的目标不是单纯提高 fuzzing 覆盖率，也不是追求一次性恢复完整状态机，而是形成一个**结构化、可追溯、可持续修正的协议分析闭环**。

## 四、研究内容

### 4.1 多源协议知识抽取

系统支持多类输入来源：

- 协议文档或 RFC 摘要
- 抓包 trace 或请求/响应日志
- 在线主动探测得到的交互记录
- 可选的协议实现代码片段

系统将这些输入统一转换为内部中间表示，便于后续分析。

### 4.2 多智能体协同分析

框架内部设置多个功能分工明确的智能体：

- **Spec Agent**：从协议文档中提取消息语义、顺序约束和字段解释
- **Trace Agent**：从抓包和交互日志中抽取消息模式、响应码、典型会话和初始状态簇
- **Code Agent**：从协议实现代码中提取 handler、分支条件、状态变量和消息处理逻辑
- **Verifier Agent**：对多个来源的结论进行一致性核验
- **Probe Agent**：针对歧义状态、冲突转移和低置信度约束生成在线探测消息

### 4.3 function calling 驱动的结构化推理

为降低纯文本推理带来的幻觉和不可控性，本课题不允许大模型直接输出最终分析结果，而是设计若干固定函数接口，由智能体通过函数调用完成分析任务。典型函数包括：

- `extract_message_types()`
- `extract_fields_and_constraints()`
- `infer_candidate_states()`
- `propose_transition()`
- `validate_transition_with_trace()`
- `generate_discriminative_probe()`
- `update_protocol_model()`

这样做可将协议分析过程转化为可审计的结构化流水线。

### 4.4 证据绑定机制

系统中的每一条协议知识均需绑定证据来源，例如：

- 文档片段
- 报文样本
- 响应模式
- 代码位置
- 在线探测结果

若某项结论缺少足够证据，则只记为 hypothesis，而不直接纳入最终协议模型。

### 4.5 在线自适应修正机制

系统先根据已有数据构建粗粒度候选状态机，再通过在线探测逐步修正模型。例如：

- 当两个状态无法有效区分时，Probe Agent 设计一条具有判别性的测试消息
- 当字段约束存在冲突时，系统生成对照输入进行验证
- 当某条状态转移只有单一证据时，系统补充主动交互以确认其有效性

### 4.6 可视化分析与交互式展示

通过 WebUI 实现以下展示：

- 协议概览与分析统计
- 协议状态机图
- 消息与字段结构表
- 证据链视图
- 在线探测与模型修正历史

## 五、拟解决的关键问题

### 5.1 如何降低协议分析中的模型幻觉

大模型在面对不完整协议说明和复杂交互序列时，容易生成听起来合理但无依据的推断。本课题通过 function calling 约束输出形式，通过 Verifier Agent 和证据绑定机制控制不可靠结论进入最终模型。

### 5.2 如何从多源异构数据中恢复协议行为知识

文档、trace、代码和在线交互的表达形式不同，如何统一表示并在不同来源间建立对应关系，是本课题的核心问题之一。

### 5.3 如何在不完整分析结果下仍输出有价值结论

完整状态机恢复难度高，尤其是复杂协议或有限时间条件下。本课题将状态机恢复与协议不变量挖掘并行进行，以保证即便状态机不完整，也能输出顺序依赖、字段约束和关键协议规则。

### 5.4 如何构建可解释的协议分析过程

单纯给出一张状态图不足以支撑可信分析。本课题要求每条状态转移、每个字段约束和每条协议规则都可追溯至证据来源，并通过 WebUI 交互式展示。

## 六、创新点

### 创新点 1：面向通用协议的 function calling 分析流水线

现有不少 AI 协议分析方法侧重直接利用大模型生成报文、推断状态机或辅助 fuzzing。本课题将协议分析任务拆分为固定函数调用流程，将模型限制在受控工具接口内，强调过程可复现、可审计、可调试。

### 创新点 2：多智能体协同的协议知识恢复

系统不依赖单一模型一次性输出结论，而是通过 Spec Agent、Trace Agent、Code Agent、Verifier Agent 和 Probe Agent 的角色分工，让协议知识由多来源协商产生。

### 创新点 3：证据绑定的协议模型构建

每条协议知识都必须与文档、trace、代码或在线探测证据绑定，并按照证据强度分为 hypothesis、supported 和 disputed 三类，提升分析结果可信度。

### 创新点 4：在线自适应修正机制

不同于纯离线分析，本课题引入主动探测环节，通过信息增益导向的 probing 策略逐步修正状态图和字段约束。

### 创新点 5：分析过程与证据链的可视化展示

不仅展示最终状态图，还展示结论形成过程、证据来源和在线修正历史，增强系统可解释性和展示效果。

## 七、总体方案设计

### 7.1 系统总体架构

系统由六个层次组成：

1. **输入适配层**：接收文档、trace、日志、代码等多源输入并转化为统一表示
2. **工具调用层**：提供消息抽取、字段约束推断、状态推断、证据评分、probe 生成等函数接口
3. **多智能体协作层**：各 agent 基于各自职责调用工具完成分析任务
4. **协议模型层**：维护候选状态机、字段约束和协议不变量
5. **在线探测层**：执行主动测试输入，获取新的行为观测
6. **WebUI 展示层**：提供协议图谱、证据链、探测历史等可视化页面

### 7.2 数据流设计

输入数据经过预处理后形成中间表示，随后被多个 agent 读取。各 agent 调用工具形成局部结论，再由 Verifier Agent 进行证据核验。核验通过的结果进入协议模型。对低置信度或冲突结论，由 Probe Agent 生成新的测试消息，在线执行后再更新模型。最终分析结果在 WebUI 中展示。

### 7.3 协议模型设计

系统内部维护两个核心对象：

#### 候选状态机模型

```json
{
  "states": ["INIT", "AUTH_PENDING", "AUTHED"],
  "transitions": [
    {
      "from": "INIT",
      "message": "USER",
      "to": "AUTH_PENDING",
      "confidence": 0.82,
      "evidence": ["trace:12", "spec:4.2"]
    }
  ]
}
```

#### 协议不变量模型

```json
{
  "invariants": [
    {
      "rule": "PASS must appear after USER",
      "type": "ordering",
      "confidence": 0.93,
      "evidence": ["trace:4", "spec:3.1"]
    }
  ]
}
```

## 八、技术路线

### 8.1 协议输入预处理

- 从 pcap 提取应用层请求/响应序列
- 对文本协议进行按消息切分和字段 token 化
- 对协议文档做章节切分和规则抽取
- 对可选代码做 handler 和状态变量定位

### 8.2 初始协议模型构建

- 基于文档和 trace 提取消息类型
- 基于会话序列聚类构建粗粒度状态簇
- 抽取初始顺序约束和字段依赖

### 8.3 证据核验与冲突管理

- 对每条候选结论记录证据来源和置信度
- 单源结论记为 hypothesis
- 双源及以上支持的结论记为 supported
- 存在冲突证据的结论记为 disputed

### 8.4 在线探测与模型修正

- 对 disputed 或低置信度结论生成判别输入
- 在线执行 probe，采集响应
- 对状态、转移和约束做增删修正
- 保留每次修正的版本历史

### 8.5 可视化输出

- 绘制协议状态图
- 展示消息结构和字段约束表
- 展示结论与证据的映射关系
- 展示 probe 执行历史和模型变更记录

## 九、实验方案

### 9.1 实验对象

为控制项目复杂度，优先选择状态性较明显、消息格式较清晰的通用协议：

- FTP
- SMTP
- POP3
- RTSP

HTTP 可作为扩展实验对象，只分析其一个简化子集。

### 9.2 实验设置

建议设置三类实验：

#### 实验一：协议知识抽取效果评估

评估消息类型识别、字段约束提取和顺序依赖挖掘的准确性。

#### 实验二：状态机恢复与在线修正效果评估

评估候选状态图质量，以及引入在线 probe 前后的修正收益。

#### 实验三：消融实验

对以下方案进行对比：

- 单 agent 自由文本分析
- 多 agent 但无证据绑定
- 多 agent + 证据绑定
- 多 agent + 证据绑定 + 在线修正完整方案

### 9.3 评估指标

- 消息类型识别准确率
- 字段约束识别准确率
- 状态转移 precision / recall
- 协议不变量正确率
- disputed 结论比例
- 在线 probe 后状态歧义下降比例
- 结论平均证据数

## 十一、可行性分析

### 11.1 技术可行性

- 文本协议如 FTP、SMTP 的消息边界清晰，便于构建最小可行系统
- 现有大模型具备文档理解、结构化抽取和函数调用能力
- Web 开发、图可视化和在线交互模块技术门槛可控

### 11.2 工作量可控性

最小版本可以只做：

- 一个协议，例如 FTP
- 三类输入：文档摘要、trace、在线交互
- 四个 agent：Spec、Trace、Verifier、Probe
- 三个前端页面：总览、状态图、证据链

在此基础上，再逐步加入 Code Agent 和第二协议实验。

### 11.3 风险与应对

**风险 1：状态机恢复不完整**  
应对：将协议不变量和字段约束作为并行输出，不以完整状态机作为唯一成功标准。

**风险 2：大模型输出不稳定**  
应对：采用 function calling 和证据绑定机制，限制自然语言直出。

**风险 3：在线探测复杂度高**  
应对：只针对少量 disputed 结论进行判别性 probing，不做大规模 fuzzing。

## 十四、结论

本课题面向 HTTP、FTP、SMTP 等通用网络协议，提出一个多智能体、证据绑定、在线自适应的协议分析框架。与已有主要面向 grammar 生成或单次状态机推断的方案相比，本课题更强调协议分析过程的结构化、可信性和可修正性。通过引入 function calling、证据链管理、在线判别探测和 WebUI 展示，该框架有望形成一个适合课堂项目实现、又具备一定研究新意的完整系统。
